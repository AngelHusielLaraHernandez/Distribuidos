    \subsection{Que és IDL, XDR, y Protocol Buffers}

    \subsubsection*{IDL (Lenguaje de Definición de Interfaces)}
    Un Lenguaje de Definición de Interfaces (IDL) es más que un simple formato de texto, es como un contrato entre diferentes sistemas informáticos. En los sistemas distribuidos donde un programa en Python en una computadora necesita pedirle datos a un servidor en C++ en otra computadora, ambos sistemas hablan idiomas diferentes en la memoria. El IDL sirve como un puente universal que describe qué funciones están disponibles para llamarse a través de la red y qué forma exacta tienen los datos que van a viajar.

    El verdadero de un IDL esta en su flujo de trabajo automatizado, el cual consta de dos fases principales:
    \begin{enumerate}
      \item \textbf{Definición:} El programador escribe un archivo abstracto (por ejemplo, myspec.idl) donde define únicamente las firmas de las funciones (nombre, parámetros y tipo de retorno), sin escribir nada de lógica interna.
      \item \textbf{Compilación:} Este archivo se pasa por un "Compilador de IDL". El compilador lee el archivo neutro y genera automáticamente código fuente en los lenguajes de programación de destino.
    \end{enumerate}
    
    Este código generado se divide en dos piezas clave para la arquitectura de Llamada a Procedimiento Remoto (RPC):

    \begin{itemize}
      \item \textbf{El Client Stub (Representante del Cliente):} Es un código que se integra en la aplicación del usuario. Actúa como un proxy local; cuando el cliente llama a una función, el stub intercepta la llamada, empaqueta los datos (proceso de marshaling) y los envía por la red.
      \item \textbf{El Server Skeleton (Esqueleto del Servidor):} Es el código que se integra en el servidor. Su trabajo es escuchar la red, recibir el paquete, desempaquetarlo (proceso de unmarshaling) y pasárselo a la función real que el desarrollador escribió en el servidor.

        \label{fig:rpc_flujo}
      \end{figure}

    \end{itemize}

    \subsubsection*{XDR (Representación Externa de Datos)}
    La Representación Externa de Datos (XDR) es un estándar de serialización de datos puro y duro. Fue concebido en la década de 1980 por Sun Microsystems y fue la tecnología que hizo posible el famoso sistema de archivos en red NFS (Network File System). Hoy en día está estandarizado bajo el RFC 4506 de la IETF.

    El propósito de XDR es resolver el problema de la variedad del hardware, diferentes procesadores almacenan los números en distinto orden (conocido como endianness, como little-endian de Intel vs big-endian de Motorola) y alinean la memoria de formas distintas. Si enviáramos la memoria cruda de una computadora a otra, la segunda interpretaría basura, para solucionarlo, XDR pone unas reglas rígidas sobre cómo convertir cualquier tipo de dato a un formato universal de bytes que pueda ser entendido por cualquier máquina, sin importar su arquitectura:

    \begin{itemize}
        \item \textbf{Tipado Implícito:} A diferencia de JSON o XML que son formatos de ``tipado explícito" que envían el nombre de la variable junto con el dato, ej: ``edad": 25, XDR es de ``tipado implícito". En XDR, no viajan metadatos ni etiquetas por el cable, si no que viajan puros ceros y unos. Esto ahorra ancho de banda y tiempo de procesamiento, pero obliga a que tanto el cliente como el servidor tengan exactamente el mismo mapa mental (el archivo .x) para saber cómo leer el flujo de bytes ciego que llega.
        \begin{figure}[H]
            \centering
            \includegraphics[width=0.7\textwidth]{img/InvPrevia/1_2.png}
            \caption{Comparación entre tipado explícito e implícito en serialización de datos.}
            \label{fig:xdr_tipado}
        \end{figure}

        \item \textbf{La Regla de Oro de los 4 Bytes:} XDR exige que la unidad básica de todo dato transmitido sea un bloque múltiplo de 4 bytes (32 bits). Los ingenieros eligieron 4 bytes debido a que 2 bytes causaban problemas de alineación en computadoras de la época, y 8 bytes desperdiciaban demasiado espacio.
        
        \item \textbf{Relleno (Padding):} Si se quiere enviar un tipo de dato pequeño como un simple carácter de 1 byte, XDR no lo envía solo. Forzosamente le inyecta 3 bytes adicionales llenos de ceros (padding) al final para completar el bloque de 4 bytes. Esto garantiza que los datos se alineen perfectamente en la memoria de casi cualquier procesador del mundo, permitiendo que las lecturas sean extremadamente rápidas, aunque sacrifique un poco de espacio en la red.
        \begin{figure}[H]
            \centering
            \includegraphics[width=0.4\textwidth]{img/InvPrevia/1_3.png}
            \caption{Padding o relleno}
            \label{fig:imagen_1_3}
        \end{figure}
    \end{itemize}

    \subsubsection*{Protocol Buffers (Protobuf)}
    Protocol Buffers es el estándar moderno de facto para microservicios. Fue creado por Google para uso interno y luego liberado al público. Protobuf funciona como un IDL (a través de sus archivos .proto) y también es el propio algoritmo de serialización binaria. Es el motor principal detrás de gRPC, el framework de comunicación de alto rendimiento de Google.

    \begin{figure}[H]
        \centering
        \includegraphics[width=0.72\textwidth]{img/InvPrevia/1_4.png}
        \caption{Proceso de compilación en Protocol Buffers, un archivo \texttt{.proto} es procesado por el compilador \texttt{protoc} para generar clases en distintos lenguajes de programación como Python, Java, C++ y Go.}
        \label{fig:protobuf_compilacion}
    \end{figure}

    Mientras XDR se basa en bloques rígidos de tamaño fijo y posiciones exactas, Protobuf fue diseñado para ser compacto, asimétrico y elástico. Google necesitaba enviar miles de millones de mensajes entre sus servidores usando el menor ancho de banda posible, y al mismo tiempo, poder actualizar los programas añadiendo o quitando campos en los mensajes sin que los servidores viejos colapsaran al no entender los nuevos campos y lo logró mediante dos innovaciones:
    
    \begin{enumerate}
        \item \textbf{Esquema Tag-Length-Value (TLV):} En lugar de depender de que las variables lleguen en un orden estricto (como XDR), Protobuf anticipa cada variable con una pequeña ``Etiqueta'' (Tag). Esta etiqueta le dice al receptor: ``El siguiente dato es la variable número 1, y es de tipo entero". Si un servidor viejo recibe un mensaje con la etiqueta ``variable número 5" que aún no conoce, simplemente lee la longitud del dato, lo salta sin fallar, y sigue leyendo el resto.
        
        \item \textbf{Compresión de Enteros Dinámicos (Varints):} En lugar de desperdiciar 4 bytes estáticos para enviar el número 1 como hace XDR obligatoriamente, Protobuf utiliza enteros de longitud variable (Varints). El sistema manipula los bits de tal forma que los números pequeños ocupan solo 1 byte en la red, mientras que números inmensos pueden llegar a ocupar hasta 10 bytes. Para lograrlo, el bit más significativo de cada byte se usa como un semáforo. Si es 1, significa ``el número continúa en el siguiente byte" y si es 0, significa ``el número termina aquí".
    \end{enumerate}

    \subsection{Tipos de datos a usar en XDR y Protocol Buffers}

    \subsubsection*{En XDR (RFC 4506)}

    El lenguaje de XDR se asemeja bastante a C y define los siguientes tipos para su transmisión:

    \begin{itemize}
        \item \textbf{Numéricos:} Enteros de 32 bits con y sin signo (\texttt{int}, \texttt{unsigned int}), enteros largos de 64 bits (\texttt{hyper}, \texttt{unsigned hyper}), y valores de punto flotante de precisión simple y doble (\texttt{float}, \texttt{double}).
        
        \item \textbf{Lógicos:} El tipo booleano (\texttt{bool}), el cual también se transmite ocupando un bloque de 32 bits.
        
        \item \textbf{Estructurados y Compuestos:}
        \begin{itemize}
            \item Cadenas de caracteres (\texttt{string}).
            \item Datos binarios sin procesar (\texttt{opaque}), ya sea de longitud fija o variable.
            \item Colecciones (\texttt{arrays} de longitud fija o variable) y estructuras (\texttt{struct}).
            \item Uniones discriminadas (\texttt{union}): A diferencia de las uniones en C, estas asocian directamente un indicador inicial (un número entero o enumeración) llamado ``discriminante'' que indica qué tipo de estructura se transmitirá a continuación dentro de la unión.
            \item Vacío (\texttt{void}): Una cantidad de cero bytes, útil para cuando no hay entrada o salida.
        \end{itemize}
    \end{itemize}

    \subsubsection*{En Protocol Buffers (proto3)}

    El catálogo de tipos está diseñado para maximizar la eficiencia binaria:

    \begin{itemize}
        \item \textbf{Estándares de longitud variable:} \texttt{int32}, \texttt{int64}, \texttt{uint32}, \texttt{uint64}. El tamaño en bytes que ocupan en red depende de la magnitud del número (varints).
        
        \item \textbf{Optimizados para signos negativos:} \texttt{sint32} y \texttt{sint64}. Emplean la codificación ``ZigZag'' que hace que los números negativos ocupen muchísimo menos espacio que si se enviaran usando el \texttt{int32} regular.
        
        \item \textbf{De tamaño fijo:} \texttt{fixed32}, \texttt{fixed64}, \texttt{sfixed32}, \texttt{sfixed64}. Siempre ocupan 4 u 8 bytes exactos. Resultan más eficientes que los varints solo si se transmiten rutinariamente números increíblemente grandes (superiores a $2^{28}$).
        
        \item \textbf{Otros:} Valores de punto flotante (\texttt{float}, \texttt{double}), booleanos (\texttt{bool}), cadenas de texto (obligatoriamente \texttt{string} codificado en UTF-8 o ASCII), y arreglos binarios arbitrarios (\texttt{bytes}).
    \end{itemize}

    \subsection{Significado de los números en archivos .x y .proto}

    A pesar de que ambos utilizan números, el propósito arquitectónico de dichos dígitos es muy diferente en cada ecosistema.

    \subsubsection*{Números en el archivo .x (XDR / ONC RPC)}

    En el lenguaje RPCL utilizado en XDR, las solicitudes de red hacia procedimientos remotos se enrutan e identifican inequívocamente gracias a una tupla de tres números: (número de programa, número de versión, número de procedimiento).

    \begin{enumerate}
        \item \textbf{Número de Programa:} Identifica una agrupación de funciones remotas afines (la aplicación en general). Existen rangos de asignación oficiales regulados por la IANA para evitar colisiones en la red (por ejemplo, el rango hexadecimal \texttt{20000000} a \texttt{3fffffff} está destinado a ser definido libremente por el usuario).
        
        \item \textbf{Número de Versión:} Permite que un mismo programa tenga múltiples versiones conviviendo en el mismo servidor de forma simultánea. Esto ayuda a no romper la compatibilidad con clientes antiguos cuando los desarrolladores añaden nuevos procedimientos.
        
        \item \textbf{Número de Procedimiento:} Señala puntualmente la función o método individual que se va a ejecutar en el servidor remoto. Como convención del estándar, el procedimiento \texttt{0} suele ser un procedimiento nulo (NULL) que no recibe ni envía datos, usado simplemente para hacer ``ping'' y probar si el servidor está en línea.
    \end{enumerate}

    \subsubsection*{Números en el archivo .proto (Protocol Buffers)}

    En Protobuf, a cada variable declarada dentro de un mensaje se le debe asignar estrictamente un Número de Campo.

    \begin{enumerate}
        \item \textbf{Reemplazo del nombre del campo:} A diferencia de JSON o XML, que repiten el nombre de la variable en el archivo (``nombre'', ``edad'', etc.), Protobuf nunca envía el nombre de la variable en la red. En su lugar, envía únicamente el número de campo para que el receptor sepa en qué ranura encajar el dato, ahorrando enormes cantidades de ancho de banda.
        
        \item \textbf{Economía de espacio:} Los números de campo del 1 al 15 ocupan solamente un byte de almacenamiento al serializarse. Por regla de optimización, estos números de oro deben reservarse exclusivamente para las variables que más se usan en una aplicación. Los números del 16 al 2047 pasan a costar dos bytes.
        
        \item \textbf{Reglas estrictas de inmutabilidad y exclusión del 0:} Los números de campo deben ser enteros positivos. El número \texttt{0} está prohibido. Además, una vez que un número ha sido asignado a un campo y el sistema está en producción, ese número no se puede cambiar ni se debe reutilizar en caso de que el campo sea borrado. Reutilizar un número generaría ambigüedad en el descifrado, corrompiendo datos, provocando fallos en el análisis, y exponiendo potencialmente información confidencial entre sistemas que usan distintas versiones del archivo.
    \end{enumerate}

    \subsection{Diseño y Justificación de la Interfaz (IDL)}
    El archivo inicial es un .x, donde es llamado \texttt{alumnos.x},
    este archivo establece la estructura de datos y los procedimientos remotos del servidor.

    \begin{lstlisting}[style=cstyle]
    struct alumno {
        int id;
        string nombre<>;
        string apellido<>;
        int edad;
        string curso<>;
    };

    struct busqueda {
        int id;
        string nombre<>;
        string curso<>;
    };

    union resultado_busqueda switch (int status) {
        case 0:
            alumno al;
        default:
            void;
    };

    struct alumno_nodo {
        alumno al;
        alumno_nodo *sig;
    };

    typedef alumno_nodo *lista_alumnos;

    program REGISTROALUMNOS {
        version REGISTROALUMNOS_V1 {
            bool registrar_alumno(alumno) = 1;
            resultado_busqueda buscar_alumno(busqueda) = 2;
            bool actualizar_alumno(alumno) = 3;
            bool eliminar_alumno(int) = 4;
            lista_alumnos listar_alumnos(void) = 5;
        } = 1;
    } = 0x20000001;
    \end{lstlisting}

    En la estructura \texttt{alumno}, los campos son el nombre, apellido y curso se 
    definieron como \texttt{string<>}. 
    Esta parte es por que indica que son cadenas de longitud variable,
    lo cual como se vio en la investigacion previa, este metodo y forma de usar lo strings de forma variable
    optimiza la transmisión de datos porque XDR solo empaqueta los bytes reales de la cadena más su longitud,
    en lugar de reservar arreglos estáticos grandes que desperdiciarían memoria.
    
    La estructura \texttt{busqueda} se baso solo en los campos que se asignaron en la actividad,
    agrupando únicamente el id, el nombre y el curso.

    Para la parte del manejo de respuestas, se implementó la estructura condicional \texttt{union resultado\_busqueda}. 
    Esta parte utiliza una variable de estado (status) como discriminante.
    Cuando la búsqueda es exitosa, y por consiguiente el estado es 0, 
    la unión pasa los datos completos del alumno encontrado. 
    Para cualquier otro estado, entonces se pasa un vacio, un (\texttt{void}).

    Debido a que Sun-RPC no soporta originalmente la parte de listas dinámicas en C como retornos directos,
    se definió un nodo de lista enlazada llamado \texttt{alumno\_nodo}. 
    Este nodo contiene la estructura del alumno y un puntero \texttt{sig}, que es el que apunta al siguiente nodo del sigueinte alumno. 

    Por ultimo, el bloque \texttt{program} define la agrupación de todas estas llamadas. Se le asignó el número hexadecimal \texttt{0x20000001}, esto en referencia a la estructura propuesta originalmente en el documento asignado.

    \subsection{Generación, Compilación y Ejecución del Proyecto}

    El compilador de protocolos nativo procesa el archivo de interfaz para generar los esqueletos en lenguaje C. 

    \begin{lstlisting}[language=bash]
    rpcgen -a -C alumnos.x
    \end{lstlisting}

    La bandera \texttt{-a} le indica al compilador que debe generar todos los archivos auxiliares posibles, incluyendo plantillas para el cliente, servidor y un archivo Makefile. La bandera \texttt{-C} asegura que el código generado cumpla con el estándar ANSI C moderno.

    Debido a que en las distribuciones actuales de Linux las rutinas de RPC fueron extraídas de la biblioteca estándar, es obligatorio utilizar la biblioteca \texttt{libtirpc} (Transport Independent RPC). Por esto, el proceso de compilación requiere vincular explícitamente estas dependencias.

    \begin{lstlisting}[language=bash]
    gcc -c alumnos_client.c -I/usr/include/tirpc
    gcc -c alumnos_server.c -I/usr/include/tirpc
    gcc alumnos_client.o alumnos_clnt.o alumnos_xdr.o -o cliente -ltirpc
    gcc alumnos_server.o alumnos_svc.o alumnos_xdr.o -o servidor -ltirpc
    \end{lstlisting}

    Durante la compilación de los códigos fuente, se utiliza la directiva \texttt{-I/usr/include/tirpc} para que el compilador encuentre los archivos de cabecera correctos. Durante el enlazado final, la instrucción \texttt{-ltirpc} incluye los binarios de la biblioteca para resolver las funciones de red. Para ejecutar el sistema, primero se lanza el servidor en segundo plano o en su propia terminal. Posteriormente, se inicia el cliente pasándole como argumento la dirección IP del equipo donde reside el servidor, utilizando \texttt{127.0.0.1} para pruebas locales.

    \begin{lstlisting}[language=bash]
    ./servidor
    ./cliente 127.0.0.1
    \end{lstlisting}

    \subsection{Explicacion del codigo del servidor}

    El servidor se tuvieron que modificar unos archivos, en los cuales,

        char ip[NI_MAXHOST] = "desconocida";
        char puerto[NI_MAXSERV] = "0";

        if (rqstp != NULL && rqstp->rq_xprt != NULL) {
            struct netbuf *caller = &rqstp->rq_xprt->xp_rtaddr;
            if (caller->buf != NULL && caller->len > 0) {
                struct sockaddr_storage ss;
                socklen_t salen;
                size_t copiar;

                memset(&ss, 0, sizeof(ss));
                copiar = caller->len < sizeof(ss) ? caller->len : sizeof(ss);
                memcpy(&ss, caller->buf, copiar);
                salen = (socklen_t)copiar;

                if (getnameinfo((struct sockaddr *)&ss, salen, ip, sizeof(ip), puerto,
                    sizeof(puerto), NI_NUMERICHOST | NI_NUMERICSERV) != 0) {
                    strcpy(ip, "desconocida");
                    strcpy(puerto, "0");
                }
            }
        }

        printf("[RECEPCION] Peticion de %s:%s para la operacion %s\n", ip, puerto,
            operacion != NULL ? operacion : "desconocida");
        fflush(stdout);
    }

    \subsection{Planteamiento de la Solución}
    La solución propuesta consiste en un sistema distribuido basado en el paradigma de \textbf{Llamada a Procedimiento Remoto (RPC)}, utilizando el framework \textbf{gRPC}. El objetivo es manejar un registro de alumnos permitiendo operaciones de creación, consulta, actualización, eliminación y listado (CRUD).

    Se decidió guardar los datos en un archivo de texto usando formato \textbf{JSON}. Esto es más sencillo que usar una base de datos grande y complicada, y permite que el programa guarde y recupere la información fácilmente cada vez que se ejecuta, además de que es fácil de mover y mantener.


    \subsection{Diseño y Justificación del Archivo \texttt{.proto}}

El archivo \texttt{alumno.proto} funciona como un acuerdo claro entre el cliente y el servidor. Ahí se define qué datos se van a enviar y recibir, y qué acciones puede hacer el servidor. Lo importante es que estas reglas no dependen del lenguaje de programación, así que ambos lados pueden entenderse sin problema. A continuación se muestra el código base del protocolo:

\begin{lstlisting}[language=C++]

\textbf{Análisis y Justificación del Diseño:}

El diseño de este archivo \texttt{.proto} sigue buenas prácticas para crear APIs con gRPC, pero se puede entender de forma sencilla así:

\begin{itemize}
    \item \textbf{Definición de Servicios (\texttt{AlumnoService}):} Se organizaron las funciones del servidor en dos tipos:
    \begin{itemize}
        \item \textit{Operaciones Unarias:} Son las más simples. El cliente hace una petición y el servidor responde una sola vez (por ejemplo: registrar, consultar, actualizar o eliminar).
        \item \textit{Operaciones de Streaming (\texttt{stream Alumno}):} En algunos casos, el servidor no envía todo de golpe, sino que manda los datos poco a poco. Esto evita usar demasiada memoria y hace que funcione mejor cuando hay muchos registros.
    \end{itemize}
    
    \item \textbf{Patrón de Mensajes Request/Response:} En lugar de usar el mismo tipo de dato para todo, se crearon mensajes específicos para cada acción (como \texttt{RegistrarRequest} o \texttt{EliminarRequest}). Esto permite que en el futuro se puedan agregar nuevos datos sin afectar lo que ya funciona.
    
    \item \textbf{Tipado Fuerte y Ligereza:} Se usan tipos de datos simples y definidos (como números, texto o verdadero/falso). Además, gRPC convierte esta información a un formato binario, lo que hace que los datos se envíen más rápido y ocupen menos espacio que en otras opciones como REST.
\end{itemize}

\subsubsection{Las Funciones del Servidor (Servicio y Métodos RPC)}

El servicio expone 6 métodos principales, los cuales cubren el ciclo de vida completo de los datos (CRUD) y consultas avanzadas. A continuación se explica su lógica interna de lado del servidor:

\textbf{1. Registrar y Consultar (Operaciones Básicas)}
\begin{lstlisting}[language=Python]
def Registrar(self, request, context):
    nuevo_id = str(uuid.uuid4())[:8] # Generacion de ID unico
    db = leer_json()
    db[nuevo_id] = {
        "id": nuevo_id, "nombre": request.nombre,
        "apellido": request.apellido, "edad": request.edad,
        "curso": request.curso, "direccion": request.direccion
    }
    guardar_json(db)
    return alumno_pb2.RegistrarResponse(exito=True, mensaje="Alumno registrado.", id=nuevo_id)

def Consultar(self, request, context):
    db = leer_json()
    if request.id in db:
        return alumno_pb2.Alumno(**db[request.id])
    context.set_code(grpc.StatusCode.NOT_FOUND) # Manejo de error
    return alumno_pb2.Alumno()
\end{lstlisting}
\begin{itemize}
    \item \textbf{\texttt{Registrar}:} Recibe los datos crudos del estudiante. El servidor le genera un ID único mediante la librería \texttt{uuid}, lo guarda en el diccionario de la base de datos y devuelve un mensaje de confirmación con el ID asignado.
    \item \textbf{\texttt{Consultar}:} Recibe únicamente el ID. Si lo encuentra en el archivo JSON, devuelve la estructura \texttt{Alumno} completa. Si no, notifica al cliente con un código de error \texttt{NOT\_FOUND}.
\end{itemize}

\textbf{2. Actualizar y Eliminar (Modificación de Estado)}
\begin{lstlisting}[language=Python]
def Actualizar(self, request, context):
    db = leer_json()
    if request.id in db:
        db[request.id] = { # Sobrescritura del registro
            "id": request.id, "nombre": request.nombre,
            "apellido": request.apellido, "edad": request.edad,
            "curso": request.curso, "direccion": request.direccion
        }
        guardar_json(db)
        return alumno_pb2.Respuesta(exito=True, mensaje="Alumno actualizado.")
    return alumno_pb2.Respuesta(exito=False, mensaje="El ID no existe.")

def Eliminar(self, request, context):
    db = leer_json()
    if request.id in db:
        del db[request.id] # Borrado atomico de la llave en el diccionario
        guardar_json(db)
        return alumno_pb2.Respuesta(exito=True, mensaje="Alumno eliminado.")
\end{lstlisting}
\begin{itemize}
    \item \textbf{\texttt{Actualizar}:} Recibe el ID del alumno y sus nuevos datos. Sobrescribe la información antigua directamente en la memoria y guarda los cambios en el archivo persistente.
    \item \textbf{\texttt{Eliminar}:} Busca el ID y utiliza la instrucción \texttt{del} para remover todo el bloque de información asociado a ese estudiante en la base de datos.
\end{itemize}

\textbf{3. Buscar y Listar (Operaciones de Streaming)}
\begin{lstlisting}[language=Python]
def BuscarPorCriterio(self, request, context):
    db = leer_json()
    criterio = request.criterio.lower()
    campo = request.campo.lower()
    for alumno_data in db.values():
        if campo in alumno_data and criterio in str(alumno_data[campo]).lower():
            yield alumno_pb2.Alumno(**alumno_data) # Retorno en flujo (Stream)

def ListarTodos(self, request, context):
    db = leer_json()
    for alumno_data in db.values():
        yield alumno_pb2.Alumno(**alumno_data) # Retorno en flujo (Stream)
\end{lstlisting}
\begin{itemize}
    \item \textbf{\texttt{BuscarPorCriterio}:} Es una función flexible que aplica un filtro ignorando mayúsculas y minúsculas (\texttt{lower()}). Utiliza la instrucción \texttt{yield} para transmitir en flujo continuo (\textit{stream}) todos los alumnos que coincidan con la búsqueda.
    \item \textbf{\texttt{ListarTodos}:} Lee el archivo JSON completo y comienza a enviar, uno por uno y sin interrupciones, todos los registros al cliente mediante un flujo optimizado.
\end{itemize}

\subsection{Desarrollo del Lado del Servidor}

\textbf{¿Qué se pretende hacer en el lado del Servidor?}

El servidor tiene como propósito fundamental actuar como el núcleo (backend) del sistema. Sus objetivos principales son:
\begin{enumerate}
    \item \textbf{Exponer los servicios} definidos en el contrato \texttt{.proto}, escuchando peticiones en un puerto de red específico de forma ininterrumpida.
    \item \textbf{Garantizar la integridad de los datos en entornos concurrentes}. Dado que gRPC permite conexiones simultáneas, el servidor pretende manejar múltiples clientes al mismo tiempo sin que la escritura en el archivo JSON (\texttt{alumnos.json}) sufra colisiones o se corrompa.
\end{enumerate}

Para lograr esto, la infraestructura del servidor se implementó de la siguiente manera:

\subsubsection{1. Concurrencia y Persistencia Segura (I/O)}
\begin{lstlisting}[language=Python]
import threading
import json
import os

JSON_FILE = "alumnos.json"
_lock = threading.Lock() # Bloqueo para evitar colisiones

def guardar_json(datos):
    """Guarda un diccionario de Python en el archivo JSON."""
    with _lock: # Solo un hilo a la vez puede entrar aqui
        with open(JSON_FILE, "w") as f:
            json.dump(datos, f, indent=4) 
\end{lstlisting}

\textbf{Explicación Detallada del Bloque:}
\begin{itemize}
    \item \textbf{Manejo del Archivo (\texttt{JSON\_FILE}):} Se utiliza un archivo de texto plano estandarizado para persistir la información sin requerir un motor de base de datos pesado.
    \item \textbf{Prevención de Condición de Carrera (\texttt{threading.Lock()}):} En un entorno distribuido, múltiples clientes pueden intentar modificar datos al mismo tiempo. Si el Cliente A y el Cliente B intentan registrar un alumno en el mismo milisegundo, el archivo \texttt{alumnos.json} podría corromperse. La variable \texttt{\_lock} actúa como un semáforo (Mutex).
    \item \textbf{Bloque de Exclusión Mutua (\texttt{with \_lock}):} Esta instrucción garantiza la seguridad en la escritura. Si un hilo (petición de un cliente) entra a este bloque, bloquea la puerta. Cualquier otra petición que intente guardar datos quedará en pausa hasta que el primer hilo termine, manteniendo la integridad absoluta de la base de datos.
\end{itemize}

\subsubsection{2. Lógica de Negocio (Clase \texttt{AlumnoServidor})}
\begin{lstlisting}[language=Python]
import uuid
import grpc
import alumno_pb2, alumno_pb2_grpc

class AlumnoServidor(alumno_pb2_grpc.AlumnoServiceServicer):
    def Registrar(self, request, context):
        nuevo_id = str(uuid.uuid4())[:8] # Generacion de ID corto
        db = leer_json()
        db[nuevo_id] = {
            "id": nuevo_id, "nombre": request.nombre,
            "apellido": request.apellido, "edad": request.edad,
            "curso": request.curso, "direccion": request.direccion
        }
        guardar_json(db)
        return alumno_pb2.RegistrarResponse(exito=True, mensaje="Alumno registrado.", id=nuevo_id)

    def BuscarPorCriterio(self, request, context):
        db = leer_json()
        criterio = request.criterio.lower()
        campo = request.campo.lower()
        for alumno_data in db.values():
            if campo in alumno_data and criterio in str(alumno_data[campo]).lower():
                yield alumno_pb2.Alumno(**alumno_data) # Retorno en flujo (Stream)
\end{lstlisting}

\textbf{Explicación Detallada de las Funciones:}
La clase hereda de la plantilla \textit{Servicer} generada automáticamente por gRPC.
\begin{itemize}
    \item \textbf{Función \texttt{Registrar} (Petición Unaria):}
    \begin{enumerate}
        \item \textit{Generación de Identidad:} Utiliza la librería \texttt{uuid.uuid4()} para generar un identificador criptográficamente único, recortándolo a 8 caracteres (\texttt{[:8]}) para que sea más amigable para el usuario.
        \item \textit{Mapeo de Datos:} Descarga la base de datos a un diccionario en memoria (\texttt{db}), crea una nueva entrada usando el ID como llave, y le asigna los datos desempaquetados de la petición del cliente (\texttt{request.nombre}, etc.).
        \item \textit{Respuesta:} Llama a la función segura \texttt{guardar\_json()} y retorna un mensaje \texttt{RegistrarResponse} confirmando el éxito de la operación.
    \end{enumerate}

    \item \textbf{Función \texttt{BuscarPorCriterio} (Server Streaming):}
    \begin{enumerate}
        \item \textit{Normalización:} Convierte tanto el campo objetivo como la palabra clave a minúsculas (\texttt{.lower()}) para realizar búsquedas insensibles a mayúsculas, mejorando la experiencia del usuario.
        \item \textit{Transmisión Continua (\texttt{yield}):} A diferencia de funciones tradicionales que usan \texttt{return} para devolver un solo bloque de datos, esta función utiliza \texttt{yield}. Esto la convierte en un \textbf{generador}. Cada vez que el ciclo \texttt{for} encuentra una coincidencia, la emite inmediatamente por el canal gRPC hacia el cliente sin cerrar la conexión. Esto evita saturar la memoria RAM del servidor al no tener que guardar listas gigantes antes de enviarlas.
    \end{enumerate}
\end{itemize}

\subsubsection{3. Inicialización Multi-hilo}
\begin{lstlisting}[language=Python]
from concurrent import futures

def iniciar_servidor():
    # Servidor capaz de manejar 10 clientes al mismo tiempo
    servidor = grpc.server(futures.ThreadPoolExecutor(max_workers=10))
    alumno_pb2_grpc.add_AlumnoServiceServicer_to_server(AlumnoServidor(), servidor)
    servidor.add_insecure_port("[::]:50051")
    servidor.start()
    servidor.wait_for_termination()
\end{lstlisting}

\textbf{Explicación Detallada de la Inicialización:}
\begin{itemize}
    \item \textbf{El Pool de Hilos (\texttt{ThreadPoolExecutor(max\_workers=10)}):} gRPC está diseñado para alta concurrencia. En lugar de procesar peticiones en una fila lenta, el servidor crea una "piscina" de 10 trabajadores independientes. Esto le otorga la capacidad de atender y resolver hasta 10 llamadas de red simultáneas en paralelo.
    \item \textbf{El Puerto y Canal (\texttt{add\_insecure\_port("[::]:50051")}):} La directiva \texttt{[::]} le indica al servidor que escuche tráfico en todas las direcciones IPv4 e IPv6 disponibles en la máquina local. Al llamarse \textit{insecure}, indica que la conexión es de texto plano sin certificados SSL/TLS, lo cual es ideal para fases de desarrollo.
    \item \textbf{Ciclo de Vida (\texttt{wait\_for\_termination()}):} Es una función bloqueante crítica. Si no estuviera, el script de Python pasaría por la línea \texttt{start()} e inmediatamente terminaría la ejecución, apagando el servidor en un segundo. Esta instrucción mantiene el hilo principal abierto e iterando de forma infinita hasta que se detenga el proceso manualmente (ej. mediante \texttt{Ctrl+C}).
\end{itemize}

\subsection{Desarrollo del Lado del Cliente}

\textbf{¿Qué se pretende hacer en el lado del Cliente?}

El cliente es la parte que interactúa directamente con el usuario. Sus objetivos son:

\begin{enumerate}
    \item \textbf{Ocultar la complejidad de la red:} Para el usuario, todo debe sentirse como un programa normal en su computadora (con un menú en consola). El cliente se encarga de enviar las peticiones al servidor y de mostrar las respuestas sin que el usuario note lo que pasa detrás.
    
    \item \textbf{Manejo de errores:} Si ocurre un problema en la red o en el servidor, el cliente debe manejarlo correctamente y mostrar un mensaje claro, en lugar de cerrarse de forma inesperada.
\end{enumerate}

Para lograr esto, el cliente se estructuró de la siguiente manera:

\subsubsection{1. Conexión Dinámica y Canal gRPC}
\begin{lstlisting}[language=Python]
import grpc
import alumno_pb2, alumno_pb2_grpc
import sys

def ejecutar(ip_servidor):
    direccion = f"{ip_servidor}:50051"
    # Abrimos el canal de comunicacion hacia la IP indicada
    with grpc.insecure_channel(direccion) as channel:
        # El stub actua como un puente transparente hacia el servidor
        stub = alumno_pb2_grpc.AlumnoServiceStub(channel)
        # ... Inicio del bucle del menu ...
\end{lstlisting}

\textbf{Explicación Detallada del Bloque:}
\begin{itemize}
    \item \textbf{Flexibilidad de Red (\texttt{ip\_servidor}):} En lugar de quemar (\textit{hardcodear}) la IP de conexión en el código, el script está diseñado para recibir la dirección del servidor como parámetro (usualmente mediante \texttt{sys.argv} al ejecutar desde la terminal). Esto permite desplegar el cliente en cualquier máquina de la red sin modificar el código fuente.
    \item \textbf{Gestión de Conexión (\texttt{grpc.insecure\_channel}):} Se establece una tubería de red directa mediante el protocolo HTTP/2 (el estándar subyacente de gRPC). La cláusula \texttt{with} es una medida de seguridad (context manager) que asegura que, sin importar si el programa termina correctamente o falla por un error, el canal de red se cerrará de forma limpia, liberando el puerto.
    \item \textbf{El Patrón Stub (\texttt{AlumnoServiceStub}):} El \textit{Stub} es un proxy o representante local. Para el programador del cliente, \texttt{stub} parece un objeto de Python normal con funciones normales, pero internamente, es el encargado de interceptar esas llamadas, serializar los datos a código binario y enviarlos a través del canal de red hacia el servidor real.
\end{itemize}

\subsubsection{2. Empaquetado y Robustez (Manejo de Errores)}
\begin{lstlisting}[language=Python]
            elif opcion == "1":
                n = input("Nombre: ")
                # ... (captura de datos extra omitida por brevedad) ...
                
                # Empaquetamos en el Request correspondiente
                request = alumno_pb2.RegistrarRequest(nombre=n, apellido="X", edad=20, curso="Y", direccion="Z")
                try:
                    # Invocacion del procedimiento remoto
                    respuesta = stub.Registrar(request)
                    if respuesta.exito:
                        print(f"ID GENERADO: {respuesta.id}") 
                except grpc.RpcError as err:
                    # Captura de fallos de red o caidas del servidor
                    print(f"\n[!] Error de conexion: {err.details()}")
\end{lstlisting}

\textbf{Explicación Detallada de las Peticiones:}
\begin{itemize}
    \item \textbf{Construcción del Mensaje (\texttt{alumno\_pb2.RegistrarRequest}):} Antes de enviar cualquier dato, este debe encapsularse estrictamente en el formato definido en el \texttt{.proto}. gRPC valida fuertemente los tipos de datos en este punto (ej. no permite enviar un \textit{string} si el proto definió un \textit{int32}).
    \item \textbf{Invocación Síncrona (\texttt{stub.Registrar(request)}):} Al ejecutar esta línea, el hilo del cliente se bloquea temporalmente esperando la respuesta del servidor. 
    \item \textbf{Tolerancia a Fallos (\texttt{try-except grpc.RpcError}):} Esta es quizás la sección más crítica en un sistema distribuido. Si el servidor se apaga repentinamente, si el cable de red se desconecta, o si la base de datos interna falla, la librería gRPC arrojará un error de tipo \texttt{RpcError}. Al capturarlo aquí, se evita que todo el script de Python del cliente se cierre abruptamente. En su lugar, se imprime un mensaje de error detallado (\texttt{err.details()}) y el usuario es retornado al menú principal de forma segura.
\end{itemize}

\subsubsection{3. Consumo de Streaming de Datos}
\begin{lstlisting}[language=Python]
            elif opcion == "5":
                campo = input("Campo a buscar: ")
                criterio = input("Que deseas buscar?: ")
                request = alumno_pb2.BuscarRequest(campo=campo, criterio=criterio)
                
                print("\n--- Resultados ---")
                # Iteracion sobre el stream de datos proveniente del servidor
                for alumno in stub.BuscarPorCriterio(request):
                    mostrar_alumno(alumno)
\end{lstlisting}

\textbf{Explicación Detallada del Flujo:}
\begin{itemize}
    \item \textbf{El Paradigma de Recepción de Streams:} A diferencia de la petición unaria anterior (donde se recibe una sola respuesta estática), la llamada a \texttt{stub.BuscarPorCriterio(request)} devuelve un iterador de Python que está "conectado en vivo" al flujo del servidor.
    \item \textbf{Desempaquetado Asíncrono (\texttt{for alumno in ...}):} Este ciclo \texttt{for} no procesa una lista que ya está en la memoria del cliente. Por el contrario, se queda "escuchando" en el canal de red. Cada vez que el servidor emite un estudiante (usando su instrucción \texttt{yield}), el cliente recibe ese paquete, lo desempaqueta, e imprime el objeto \texttt{alumno} usando la función auxiliar \texttt{mostrar\_alumno}. Esto permite visualizar grandes volúmenes de datos en tiempo real, con una latencia mínima y sin saturar la memoria RAM de la máquina cliente.
\end{itemize}
