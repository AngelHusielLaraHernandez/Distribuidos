    La ubicación de este retardo (\texttt{time.sleep(2)}) se implementó en esa parte para que la simulación funcione correctamente. Al colocarlo justo después de haber guardado el tiempo de envío local ($t_0$) y antes de procesar la respuesta del servidor ($C_s$), se obliga al cliente a pausar su ejecución durante 2 segundos y con esto simulamos el comportamiento de una red con alta latencia donde el mensaje tarda más tiempo en viajar, lo que incrementará directamente el RTT (Round-Trip Time) calculado posteriormente.

    Al analizar ambas salidas, el cambio en el ajuste del reloj es evidente. En la ejecución sin latencia, el tiempo de viaje total (RTT) es de apenas unos microsegundos (0.000635 s), lo que resulta en un desfase prácticamente nulo. Por lo tanto, el nuevo reloj ajustado ($C$) es casi idéntico al reloj local actual del cliente.

    Por el contrario, al introducir el retardo de 2 segundos, el RTT se dispara a $\approx$ 2.0005 segundos, y el algoritmo de Cristian asume un desfase simétrico para el viaje de regreso de $\approx$ 1.0002 segundos. Dado que el servidor guardó su tiempo ($C_s$) de forma inmediata tras recibir la petición, pero el cliente tardó 2 segundos en procesarla, el cálculo final del nuevo reloj ($C$) resulta ser un segundo menor que el tiempo local actual del cliente ($C < C_c$). 

    Específicamente, el tiempo calculado es \texttt{1775026740.58}, mientras que el reloj local ya avanzó hasta \texttt{1775026741.58}. Bajo estas condiciones, las reglas de sincronización nos dicen que el reloj se debe detener o se ralentizar gradualmente por la diferencia calculada ($C_c - C$ unidades) hasta lograr la coincidencia.

    \textbf{\textcolor{blue}{Modificar la implementación anterior para que se pueda trabajar con varios clientes al mismo tiempo (máquinas diferentes, pueden ser físicas o virtuales y multithread) y se vea una diferencia de tiempo.}}

    Para lograr que el servidor atienda a múltiples clientes de manera simultánea, se modificó el script original incorporando la librería \texttt{threading}. En lugar de procesar cada conexión de manera secuencial y bloqueante, se definió una función \texttt{manejar\_cliente} y se instanció un nuevo hilo por cada conexión entrante en el ciclo principal.

    \textbf{Fragmento modificado en el Servidor (Multithread):}
    \begin{lstlisting}[language=Python]
import socket
import time
import threading # Libreria agregada para soporte multihilo

# Funcion que maneja la peticion de cada cliente de forma independiente
def manejar_cliente(conn, addr):
    print(f"[+] Hilo iniciado para conexion desde: {addr}")
    Cs = time.time()
    conn.send(str(Cs).encode())
    conn.close()
    print(f"[-] Tiempo enviado ({Cs}) a {addr}. Hilo terminado.\n")

# ... (Configuracion del socket) ...
server.listen(10) # Se permite una cola de hasta 10 conexiones simultaneas

while True:
    conn, addr = server.accept()
    # Se crea y arranca un nuevo hilo por cada cliente conectado
    hilo = threading.Thread(target=manejar_cliente, args=(conn, addr))
    hilo.start()
    \end{lstlisting}

    Por el lado del cliente, para generar una diferencia de tiempo visible entre cada ejecución, se sustituyó el retardo fijo de 2 segundos por un retardo dinámico utilizando la librería \texttt{random}.

    \textbf{\textcolor{blue}{Agregar al código lo necesario para considerar cuando C > Cc o C < Cc.}}

    Para cumplir con este punto, se agregó un bloque lógico \texttt{if / elif} al final del script del cliente. Este bloque evalúa la relación entre el nuevo tiempo calculado ($C$) y el tiempo local actual del cliente ($C_c$), calculando las diferencias exactas y determina la acción correcta según las reglas de sincronización.

    \textbf{Fragmento modificado en el Cliente (Lógica de evaluación):}
    \begin{lstlisting}[language=Python]
# --- LOGICA DE AJUSTE ---
if C > Cc:
    diferencia = C - Cc
    print("[ATENCION] C > Cc (El tiempo calculado es MAYOR al local)")
    print("         -> ACCION: Se ADELANTA el reloj local a la nueva hora.")
    print(f"        -> El salto hacia adelante es de {diferencia:.6f} segundos.")
elif C < Cc:
    diferencia = Cc - C
    print("[ATENCION] C < Cc (El tiempo calculado es MENOR al local)")
    print(f"         -> ACCION: El reloj local se debe ralentizar gradualmente.")
    print(f"         -> Se debe RALENTIZAR el reloj por {diferencia:.6f} unidades.")
else:
    print("[LISTO] C == Cc (El reloj ya esta sincronizado).")
    \end{lstlisting}

        \begin{enumerate}
            \item \textbf{¿La latencia afecta el resultado final?}

            No, la latencia \textbf{no afecta el resultado final} de la sincronización en términos de convergencia. Como se puede observar en la salida, a pesar de que cada cliente experimentó diferentes latencias (desde 0.232 hasta 2.674 segundos), todos convergieron al mismo tiempo promedio (1774828782.641636). Esto se debe a que el algoritmo de Berkeley trabaja con los \textit{valores de tiempo reportados} por cada cliente, no con el momento exacto en que esos valores llegan al servidor.

            Sin embargo, la latencia sí afecta la \textbf{precisión} del resultado. Si un cliente tiene alta latencia, el tiempo que reporta ya estará ``desactualizado'' cuando llegue al servidor, introduciendo un error sistemático. En nuestra implementación, el cliente captura su tiempo local \textit{antes} de la latencia, por lo que el ajuste calculado es correcto para ese instante, pero cuando el cliente finalmente aplica el ajuste, su reloj real ya avanzó.

            \item \textbf{¿Por qué el algoritmo de Berkeley es sensible o no a este factor?}

            El algoritmo de Berkeley es \textbf{parcialmente sensible} a la latencia por las siguientes razones:

            \textbf{Insensibilidad:} El algoritmo no depende de una fuente de tiempo externa (como NTP), sino que calcula un promedio democrático de todos los participantes. La latencia no impide que el algoritmo calcule y distribuya los ajustes correctamente.

            \textbf{Sensibilidad:} En redes con alta latencia o latencia asimétrica, el algoritmo puede introducir errores porque:
            \begin{itemize}
                \item El tiempo que tarda un mensaje en llegar no se compensa (a diferencia del algoritmo de Cristian que usa RTT/2).
                \item Si la latencia varía significativamente entre clientes, algunos recibirán sus ajustes más tarde, acumulando desfase adicional.
            \end{itemize}

            Para mitigar esto, implementaciones robustas de Berkeley incorporan estimación de RTT (Round-Trip Time) y compensan la latencia al calcular los ajustes.
        \end{enumerate}
    \end{itemize}

    import time
    import struct

    def servidorBerkeley(port=60001, num_clients=3):
        sockServer = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        sockServer.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)

        # Usar '0.0.0.0' para aceptar conexiones de cualquier IP
        sockServer.bind(('0.0.0.0', port))
        sockServer.listen(num_clients)

        # Obtener la IP local para mostrar al usuario

        local_ip = socket.gethostbyname(hostname)
        print(f"Servidor de Berkeley escuchando en {local_ip}:{port}")
        print(f"Esperando {num_clients} clientes...")

        connections = []
        times = []

        for i in range(num_clients):
            conn, addr = sockServer.accept()
            print(f"Conexión {i+1}/{num_clients} establecida con: {addr}")
            connections.append(conn)

        print("\nTodos los clientes conectados. Solicitando tiempos...")

        for conn in connections:
            conn.send(b'TIME_REQUEST')

        for conn in connections:
            client_time = struct.unpack('d', conn.recv(8))[0]
            print(f"Tiempo recibido del cliente: {client_time:.2f}")
            times.append(client_time)

        all_times = times + [server_time]
        average_time = sum(all_times) / len(all_times)
        print(f"Tiempo promedio calculado: {average_time:.2f}")

        adjusted_times = [average_time - t for t in times]
        
        # Para imprimir una lista con 2 decimales, usamos esta técnica de comprensión de listas:
        ajustes_formateados = [f"{adj:.2f}" for adj in adjusted_times]

            conn.send(struct.pack('f', adjustment))

        for conn in connections:
            conn.close()

        sockServer.close()
        print("\nSincronización completada. Servidor cerrado.")

    if __name__ == "__main__":

    El ambiente diseñado para esta simulación tiene siete procesos (P0 a P6) que interactúan de la siguiente manera, asegurando que el reloj de Lamport siga funcionando bien cuando el sistema crece, incluso si las conexiones entre los nodos no son simples o en línea recta.

\begin{itemize}
    \item \textbf{P0:} Inicia la cascada ejecutando un evento local y enviando un mensaje a P1.
    \item \textbf{P1:} Recibe el mensaje de P0, procesa un evento local, y envía mensajes hacia dos procesos distintos: P2 y P3.
    \item \textbf{P2 y P3:} Reciben los mensajes de P1 de manera paralela. P2 envía su mensaje a P4, mientras que P3 envía su mensaje a P5.
    \item \textbf{P4 y P5:} Reciben la información de sus respectivos emisores y la reenvían hacia el nodo final, P6.
    \item \textbf{P6:} Actúa como nodo colector. Recibe los mensajes tanto de P4 como de P5, actualizando su reloj de Lamport según las estampas de tiempo recibidas, y finaliza con un evento local.
\end{itemize}

from multiprocessing import Process, Pipe
from os import getpid
from datetime import datetime

# ==========================================
# CÓDIGOS DE COLOR ANSI PARA LA TERMINAL
# ==========================================
RESET = "\033[0m"
C_EVENT = "\033[93m"  
C_SEND  = "\033[92m"  
C_RECV  = "\033[96m"  
C_TIME  = "\033[90m"  

def local_time(counter):
    time_str = datetime.now().strftime('%H:%M:%S.%f')[:-3]
    return f'{C_TIME}[ Reloj Lamport: {counter:02d} | Hora: {time_str} ]{RESET}'

def calc_recv_timestamp(recv_time_stamp, counter):
    return max(recv_time_stamp, counter) + 1

def event(pid, counter, p_num):
    counter += 1
    accion = f'{C_EVENT}[P{p_num}] EVENTO LOCAL   (PID:{pid}){RESET}'.ljust(52)
    print(f'{accion} {local_time(counter)}')
    return counter

def send_message(pipe, pid, counter, p_num, dest_num):
    counter += 1
    pipe.send(('Mensaje', counter))
    accion = f'{C_SEND}[P{p_num}] ENVÍA a P{dest_num}    (PID:{pid}){RESET}'.ljust(52)
    print(f'{accion} {local_time(counter)}')
    return counter

def recv_message(pipe, pid, counter, p_num, src_num):
    message, timestamp = pipe.recv()
    counter = calc_recv_timestamp(timestamp, counter)
    accion = f'{C_RECV}[P{p_num}] RECIBE de P{src_num}  (PID:{pid}){RESET}'.ljust(52)
    print(f'{accion} {local_time(counter)}')
    return counter

# ==========================================
# DEFINICIÓN DE LOS 7 PROCESOS
# ==========================================
def p0(pipe01):
    pid = getpid()
    counter = 0
    counter = event(pid, counter, 0)
    counter = send_message(pipe01, pid, counter, 0, 1)

def p1(pipe10, pipe12, pipe13):
    pid = getpid()
    counter = 0
    counter = recv_message(pipe10, pid, counter, 1, 0)
    counter = event(pid, counter, 1)
    counter = send_message(pipe12, pid, counter, 1, 2)
    counter = send_message(pipe13, pid, counter, 1, 3)

def p2(pipe21, pipe24):
    pid = getpid()
    counter = 0
    counter = recv_message(pipe21, pid, counter, 2, 1)
    counter = send_message(pipe24, pid, counter, 2, 4)

def p3(pipe31, pipe35):
    pid = getpid()
    counter = 0
    counter = recv_message(pipe31, pid, counter, 3, 1)
    counter = send_message(pipe35, pid, counter, 3, 5)

def p4(pipe42, pipe46):
    pid = getpid()
    counter = 0
    counter = recv_message(pipe42, pid, counter, 4, 2)
    counter = send_message(pipe46, pid, counter, 4, 6)

def p5(pipe53, pipe56):
    pid = getpid()
    counter = 0
    counter = recv_message(pipe53, pid, counter, 5, 3)
    counter = send_message(pipe56, pid, counter, 5, 6)

def p6(pipe64, pipe65):
    pid = getpid()
    counter = 0
    counter = recv_message(pipe64, pid, counter, 6, 4)
    counter = recv_message(pipe65, pid, counter, 6, 5)
    counter = event(pid, counter, 6)

    print("\n" + "="*70)
    print(" INICIANDO SIMULACIÓN DE RELOJES DE LAMPORT (7 PROCESOS) ".center(70, "="))
    print("="*70 + "\n")

    # Creación de tuberías para comunicación

    pipe24, pipe42 = Pipe()
    pipe35, pipe53 = Pipe()
    pipe46, pipe64 = Pipe()
    pipe56, pipe65 = Pipe()

        Process(target=p2, args=(pipe21, pipe24)),
        Process(target=p3, args=(pipe31, pipe35)),
        Process(target=p4, args=(pipe42, pipe46)),
        Process(target=p5, args=(pipe53, pipe56)),

        p.start()
    for p in procesos:
        p.join()
        
    print("\n" + "="*70)
    print(" SIMULACIÓN FINALIZADA ".center(70, "="))
    print("="*70 + "\n")

El código implementa el algoritmo de sincronización de Lamport usando la librería \texttt{multiprocessing} de Python, la cual permite ejecutar los 7 procesos de manera verdaderamente concurrente, cada uno con su propio espacio de memoria y su propio Identificador de Proceso (PID).

\textbf {1. Reglas Lógicas del Reloj de Lamport}

El núcleo del algoritmo se divide en tres funciones fundamentales que alteran la variable \texttt{counter} (el reloj lógico local de cada proceso):
\begin{itemize}
    \item \textbf{Eventos Locales (\texttt{event}):} Simplemente incrementan el reloj local en 1 unidad ($C_i = C_i + 1$).
    \item \textbf{Envío de Mensajes (\texttt{send\_message}):} Antes de enviar, el proceso incrementa su reloj local en 1 ($C_i = C_i + 1$) y luego adjunta ese valor actualizado (estampa de tiempo) dentro de la tubería junto con el mensaje mediante \texttt{pipe.send(('Mensaje', counter))}.
    \item \textbf{Recepción de Mensajes (\texttt{recv\_message}):} Cuando un proceso extrae un mensaje de la tubería mediante \texttt{pipe.recv()}, lee la estampa de tiempo adjunta. Es aquí donde se aplica la regla de convergencia principal de Lamport implementada en la función \texttt{calc\_recv\_timestamp}: el nuevo tiempo del proceso receptor será el valor máximo entre su propio reloj local y la estampa de tiempo recibida del mensaje, más uno ($C_j = \max(C_j, C_m) + 1$).
\end{itemize}

\textbf{2. Cumplimeiento del ambiente propuesto}

La comunicación entre procesos se logra mediante objetos \texttt{Pipe()}, que actúan como canales bidireccionales. Para garantizar que el programa se comporte exactamente como el diagrama porpuesto, las conexiones y llamadas a las funciones se estructuraron de la siguiente forma:
\begin{itemize}
    \item \textbf{El Inicio (P0):} La función \texttt{p0} arranca el sistema ejecutando un \texttt{event()} seguido de un \texttt{send\_message()} usando \texttt{pipe01}. Esto coincide con el origen de la cascada.
    \item \textbf{La Ramificación (P1):} La función \texttt{p1} está diseñada para bloquearse hasta recibir el mensaje de P0 mediante \texttt{recv\_message(pipe10)}. Una vez que el reloj se actualiza, P1 ejecuta un evento local y luego se divide, llamando a \texttt{send\_message} dos veces: una hacia P2 (\texttt{pipe12}) y otra hacia P3 (\texttt{pipe13}). 
    \item \textbf{El Paralelismo (P2, P3, P4, P5):} Las funciones \texttt{p2} y \texttt{p3} funcionan como intermediarios en paralelo. Al recibir el mensaje de P1, envían su propio mensaje hacia P4 y P5 respectivamente. A su vez, \texttt{p4} y \texttt{p5} reciben esos mensajes y los retransmiten hacia el colector final. Esto asegura que la estampa de tiempo siga arrastrando la historia causal de los eventos previos de manera paralela.
    \item \textbf{La Convergencia (P6):} La función \texttt{p6} consolida la información de las dos ramas. Realiza dos llamadas a \texttt{recv\_message}, extrayendo los mensajes de \texttt{pipe64} y \texttt{pipe65}. La función \texttt{calc\_recv\_timestamp} internamente se asegurará de que el reloj de P6 absorba el tiempo más grande que le llegue de ambas ramas. Finalmente, realiza un último \texttt{event()} para marcar la conclusión del flujo. 
\end{itemize}

Este diseño refleja fielmente las flechas trazadas en el diagrama de ambiente propuesto, forzando a los procesos a respetar la relación ``sucedió antes que'' ($\rightarrow$) diseñada para esta topología de 7 nodos y validando el correcto funcionamiento de los relojes lógicos.

\textbf{1. Reglas Lógicas del Reloj Vectorial}

Cada proceso inicia con un arreglo de ceros: \texttt{[0, 0, 0, 0, 0]}. La posición \texttt{i} del arreglo corresponde al conocimiento que tiene el proceso sobre el tiempo del proceso \texttt{i}.
\begin{itemize}
    \item \textbf{Eventos Locales y Envíos:} Cada vez que un proceso \texttt{Pi} hace un evento o envía un mensaje, incrementa únicamente su propia posición en el vector (\texttt{counter[i] += 1}). En el caso de envío, manda el arreglo completo por la tubería.
    \item \textbf{Recepción de Mensajes:} Cuando \texttt{Pi} recibe un mensaje, compara su vector actual con el vector recibido. Usa la función \texttt{max()} \textbf{posición por posición} para quedarse con el valor más grande de cada índice. Finalmente, incrementa su propio índice para registrar el evento de recepción.
\end{itemize}

\textbf{2. Cumplimiento del ambiente propuesto}

La estructura de las tuberías (\texttt{Pipe}) sigue fielmente el diagrama diseñado:
\begin{itemize}
    \item \textbf{P0} tiene su vector inicial y tras sus tres acciones (un evento local y dos envíos) su índice 0 avanza, propagando esos valores a P1 y P2.
    \item \textbf{P1 y P2} reciben la historia de P0. Al actualizar sus vectores, adoptan el valor del índice 0 que traía P0, y luego incrementan sus respectivos índices (1 y 2) para retransmitirlos a los siguientes procesos.
    \item \textbf{P3} actúa como puente, absorbiendo la historia de P1 (que ya incluía a P0) y enviándola a P4.
    \item \textbf{P4 (Convergencia):} Es el punto clave de la simulación. Al recibir información de P2 y de P3, su función \texttt{calc\_recv\_timestamp} combina ambas historias usando los máximos. Esto le permite a P4 tener una ``fotografía'' completa de cuántos eventos ocurrieron en cada uno de los procesos anteriores.
\end{itemize}

\textbf{Funcionamiento del código}

Al analizar los resultados de la consola , podemos comprobar paso a paso cómo los relojes vectoriales preservan toda la historia causal. A diferencia de Lamport, que solo daba un número general, aquí podemos ver exactamente qué sabe cada proceso sobre el resto del sistema.

Observando la ejecución, el proceso \textbf{P0} inicia la cascada: tras su primer evento local y sus envíos a P1 y P2, vemos cómo únicamente incrementa su propia posición (la posición 0 del arreglo), llegando hasta el vector \texttt{[03, 00, 00, 00, 00]}. 

Cuando los demás procesos reciben mensajes, podemos notar cómo absorben esta historia. Por ejemplo, cuando \textbf{P1} recibe el mensaje de P0, compara el vector recibido con el suyo y actualiza su reloj a \texttt{[02, 01, 00, 00, 00]}, adoptando la información de los 2 eventos que llevaba P0 en ese instante y sumando el suyo propio. De igual forma, cuando \textbf{P2} recibe su mensaje, su vector se actualiza a \texttt{[03, 00, 01, 00, 00]}.

La ventaja principal de este algoritmo se demuestra en el nodo de convergencia, \textbf{P4}. En la imagen, vemos que P4 recibe primero el mensaje de P2, actualizando su vector a \texttt{[03, 00, 02, 00, 01]}. Inmediatamente después, recibe el mensaje de P3 (el cual trae un vector de \texttt{[02, 02, 00, 02, 00]}). Al aplicar la regla de tomar el valor máximo posición por posición entre ambos vectores, P4 fusiona exitosamente ambas ramas. 

Tras su evento local final, P4 concluye con el vector \texttt{[03, 02, 02, 02, 03]}. Este arreglo final nos dice con exactitud que, antes de que P4 terminara, ocurrieron 3 eventos en P0, 2 en P1, 2 en P2 y 2 en P3. Esto demuestra que la topología funcionó perfectamente y que no hubo ninguna pérdida de información en todo el sistema.
