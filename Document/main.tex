    \begin{verbatim}
        delay = 2 # segundos

    Este fragmento se implementó en el script del cliente en la siguiente ubicación exacta:

    \begin{lstlisting}[language=Python]
# 2. Registrar el tiempo exacto antes de enviar la solicitud (t0)
t0 = time.time()
# Actividad 1: Simular latencia en red de 2 segundos

    La ubicación de este retardo (\texttt{time.sleep(2)}) se implementó en esa parte para que la simulación funcione correctamente. Al colocarlo justo después de haber guardado el tiempo de envío local ($t_0$) y antes de procesar la respuesta del servidor ($C_s$), se obliga al cliente a pausar su ejecución durante 2 segundos y con esto simulamos el comportamiento de una red con alta latencia donde el mensaje tarda más tiempo en viajar, lo que incrementará directamente el RTT (Round-Trip Time) calculado posteriormente.

    \textbf{\textcolor{blue}{¿Como cambia el ajuste del reloj?}}

    Antes de responder a la pregunta debemos observar el comportamiento de la simulación mediante las salidas en consola, tanto en una ejecución normal (sin latencia) como en la simulación con latencia, como se muestra a continuación:

    Al analizar ambas salidas, el cambio en el ajuste del reloj es evidente. En la ejecución sin latencia, el tiempo de viaje total (RTT) es de apenas unos microsegundos (0.000635 s), lo que resulta en un desfase prácticamente nulo. Por lo tanto, el nuevo reloj ajustado ($C$) es casi idéntico al reloj local actual del cliente.

    Por el contrario, al introducir el retardo de 2 segundos, el RTT se dispara a $\approx$ 2.0005 segundos, y el algoritmo de Cristian asume un desfase simétrico para el viaje de regreso de $\approx$ 1.0002 segundos. Dado que el servidor guardó su tiempo ($C_s$) de forma inmediata tras recibir la petición, pero el cliente tardó 2 segundos en procesarla, el cálculo final del nuevo reloj ($C$) resulta ser un segundo menor que el tiempo local actual del cliente ($C < C_c$). 

    Específicamente, el tiempo calculado es \texttt{1775026740.58}, mientras que el reloj local ya avanzó hasta \texttt{1775026741.58}. Bajo estas condiciones, las reglas de sincronización nos dicen que el reloj se debe detener o se ralentizar gradualmente por la diferencia calculada ($C_c - C$ unidades) hasta lograr la coincidencia.

    \textbf{\textcolor{blue}{Modificar la implementación anterior para que se pueda trabajar con varios clientes al mismo tiempo (máquinas diferentes, pueden ser físicas o virtuales y multithread) y se vea una diferencia de tiempo.}}

    Para lograr que el servidor atienda a múltiples clientes de manera simultánea, se modificó el script original incorporando la librería \texttt{threading}. En lugar de procesar cada conexión de manera secuencial y bloqueante, se definió una función \texttt{manejar\_cliente} y se instanció un nuevo hilo por cada conexión entrante en el ciclo principal.

    \textbf{Fragmento modificado en el Servidor (Multithread):}
    \begin{lstlisting}[language=Python]
import socket
import time
import threading # Libreria agregada para soporte multihilo

# Funcion que maneja la peticion de cada cliente de forma independiente
def manejar_cliente(conn, addr):
    print(f"[+] Hilo iniciado para conexion desde: {addr}")
    Cs = time.time()
    conn.send(str(Cs).encode())
    conn.close()
    print(f"[-] Tiempo enviado ({Cs}) a {addr}. Hilo terminado.\n")

# ... (Configuracion del socket) ...
server.listen(10) # Se permite una cola de hasta 10 conexiones simultaneas

while True:
    conn, addr = server.accept()
    # Se crea y arranca un nuevo hilo por cada cliente conectado
    hilo = threading.Thread(target=manejar_cliente, args=(conn, addr))
    hilo.start()
    \end{lstlisting}

    Por el lado del cliente, para generar una diferencia de tiempo visible entre cada ejecución, se sustituyó el retardo fijo de 2 segundos por un retardo dinámico utilizando la librería \texttt{random}.

    \textbf{Fragmento modificado en el Cliente (Latencia aleatoria):}
    \begin{lstlisting}[language=Python]
import random

# Simulamos una latencia aleatoria para cada cliente (entre 0.1 y 2.5 segundos)
delay = random.uniform(0.1, 2.5)
time.sleep(delay)
    \end{lstlisting}

    \vspace{0.3cm}
    \textbf{\textcolor{blue}{Agregar al código lo necesario para considerar cuando C > Cc o C < Cc.}}

    Para cumplir con este punto, se agregó un bloque lógico \texttt{if / elif} al final del script del cliente. Este bloque evalúa la relación entre el nuevo tiempo calculado ($C$) y el tiempo local actual del cliente ($C_c$), calculando las diferencias exactas y determina la acción correcta según las reglas de sincronización.

    \textbf{Fragmento modificado en el Cliente (Lógica de evaluación):}
    \begin{lstlisting}[language=Python]
# --- LOGICA DE AJUSTE ---
if C > Cc:
    diferencia = C - Cc
    print("[ATENCION] C > Cc (El tiempo calculado es MAYOR al local)")
    print("         -> ACCION: Se ADELANTA el reloj local a la nueva hora.")
    print(f"        -> El salto hacia adelante es de {diferencia:.6f} segundos.")
elif C < Cc:
    diferencia = Cc - C
    print("[ATENCION] C < Cc (El tiempo calculado es MENOR al local)")
    print(f"         -> ACCION: El reloj local se debe ralentizar gradualmente.")
    print(f"         -> Se debe RALENTIZAR el reloj por {diferencia:.6f} unidades.")
else:
    print("[LISTO] C == Cc (El reloj ya esta sincronizado).")
    \end{lstlisting}

    Como se vio en la prueba anterior, si los relojes locales ya están en hora con el servidor pero hay latencia de red, siempre se da el caso de $C < C_c$. Para demostrar que el código también considera y evalúa correctamente el caso opuesto ($C > C_c$), se implementó un mecanismo para simular que el cliente tiene su reloj interno atrasado 10 segundos respecto a la realidad.

    \textbf{Respuestas:}
    \begin{enumerate}
        \item \textbf{¿La latencia afecta el resultado final?}

        No, la latencia \textbf{no afecta el resultado final} de la sincronización en términos de convergencia. Como se puede observar en la salida, a pesar de que cada cliente experimentó diferentes latencias (desde 0.232 hasta 2.674 segundos), todos convergieron al mismo tiempo promedio (1774828782.641636). Esto se debe a que el algoritmo de Berkeley trabaja con los \textit{valores de tiempo reportados} por cada cliente, no con el momento exacto en que esos valores llegan al servidor.

        Sin embargo, la latencia sí afecta la \textbf{precisión} del resultado. Si un cliente tiene alta latencia, el tiempo que reporta ya estará ``desactualizado'' cuando llegue al servidor, introduciendo un error sistemático. En nuestra implementación, el cliente captura su tiempo local \textit{antes} de la latencia, por lo que el ajuste calculado es correcto para ese instante, pero cuando el cliente finalmente aplica el ajuste, su reloj real ya avanzó.

        \item \textbf{¿Por qué el algoritmo de Berkeley es sensible o no a este factor?}

        El algoritmo de Berkeley es \textbf{parcialmente sensible} a la latencia por las siguientes razones:

        \textbf{Insensibilidad:} El algoritmo no depende de una fuente de tiempo externa (como NTP), sino que calcula un promedio democrático de todos los participantes. La latencia no impide que el algoritmo calcule y distribuya los ajustes correctamente.

        \textbf{Sensibilidad:} En redes con alta latencia o latencia asimétrica, el algoritmo puede introducir errores porque:
        \begin{itemize}
            \item El tiempo que tarda un mensaje en llegar no se compensa (a diferencia del algoritmo de Cristian que usa RTT/2).
            \item Si la latencia varía significativamente entre clientes, algunos recibirán sus ajustes más tarde, acumulando desfase adicional.
        \end{itemize}

        Para mitigar esto, implementaciones robustas de Berkeley incorporan estimación de RTT (Round-Trip Time) y compensan la latencia al calcular los ajustes.
    \end{enumerate}
